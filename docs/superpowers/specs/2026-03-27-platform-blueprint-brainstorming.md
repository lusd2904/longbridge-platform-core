# 2026-03-27 项目总体蓝图脑暴记录

## 目标

为 `refactor-v2` 形成一份覆盖功能、产品结构、设计、UI 与编码治理的总体蓝图，并在脑暴过程中持续沉淀关键决策。

## 项目现状摘要

- 当前项目是平台化重构，不再是单页 demo。
- 现有结构已经分为 `web-portal`、`api-gateway` 和多个业务服务。
- 前端路由已经呈现出多个子系统：`workspace`、`market`、`trading`、`analysis`、`platform`。
- 当前讨论目标不是直接实现，而是先统一产品与系统蓝图。

## 已确认决策

### 1. 本轮输出范围

- 选择：`A 总体蓝图`
- 说明：先统一全局方向，覆盖功能分层、用户角色、信息架构、设计语言、UI 体系、技术分层与重构路线。

### 2. 用户与角色方向

- 结论：产品需要覆盖多类用户，而不是只服务单一角色。
- 用户补充要求：需要通过用户权限配置不同角色，角色拥有各自菜单。
- 影响：
  - 产品必须是平台型而不是单工作流工具。
  - 导航和首页不能写死为单一路径。
  - 菜单体系必须由权限模型驱动。

### 3. 权限与菜单模型

- 选择：`C 完全自定义 RBAC`
- 说明：角色、菜单、按钮、数据范围都应支持独立配置。
- 影响：
  - 权限系统是核心基础设施，不是附属功能。
  - 前端菜单、页面访问、操作权限、后端接口授权必须统一建模。

### 4. 用户与角色关系

- 选择：`C 单用户多角色，并支持工作台视角切换`
- 说明：同一用户可拥有多个角色，并可按当前视角切换看到对应菜单与工作台。
- 影响：
  - 产品需要“视角层”而不只是“权限层”。
  - 首页、导航、快捷操作、最近访问和默认落点都要与当前视角绑定。

### 5. 终端策略

- 选择：`C 三端并重`
- 说明：Web、桌面端、移动端从一开始就统一规划，而不是后续补适配。
- 影响：
  - 设计系统要先定义跨端规则。
  - 权限、导航、信息密度、交互层级需要支持多终端映射。

### 6. 组织形态

- 选择：`D 先按单企业平台设计，但预留多租户扩展`
- 说明：当前优先服务单企业内部多岗位协作，同时为未来 SaaS 化保留扩展位。
- 影响：
  - 组织、部门、角色、用户、菜单、权限的模型需要避免写死。
  - API 和数据模型应预留 `tenant/org` 扩展位，但不提前引入过重复杂度。

### 7. 权限控制粒度

- 选择：`C 菜单 + 页面 + 按钮/操作 + 数据范围`
- 说明：不仅要控制可见页面，也要控制具体动作以及可访问的数据边界。
- 影响：
  - 系统蓝图必须覆盖数据权限。
  - 后端服务间需要统一鉴权上下文。
  - 前端需要处理“可见但只读”“可见但不可执行”“仅特定数据域可见”等状态。

### 8. 登录后默认进入方式

- 选择：`B 按当前视角进入对应工作台`
- 说明：用户登录后应先进入当前选定视角对应的工作台，而不是统一总控首页。
- 影响：
  - 平台需要维护“当前视角”和“默认视角”。
  - 首页信息架构需要按视角定制，而不是所有角色共用一套首页。
  - 三端的启动页、底部导航、侧边栏和快捷入口都应围绕视角展开。

### 9. 角色与权限维护方式

- 选择：`B 平台超级管理员在后台维护`
- 说明：角色、菜单、按钮权限和数据范围应主要由平台超级管理员通过后台进行配置与维护。
- 影响：
  - 平台侧需要独立的权限治理后台，而不是仅靠代码内配置。
  - 角色模板、菜单树、权限点、数据范围规则需要可视化管理。
  - 权限变更应支持审计与可回溯，以避免误配置带来业务风险。

### 10. 治理强度

- 选择：`C 审计 + 审批流 + 留痕`
- 说明：高风险操作需要具备审计记录、审批/复核能力、操作原因记录和结果追踪。
- 影响：
  - 权限治理、交易执行、配置变更等域都应纳入统一治理模型。
  - 产品蓝图中需要单独规划审批流、审计日志、操作回放或追踪能力。
  - 编码与服务设计层需要为关键事件保留标准化事件记录和状态流转模型。

### 11. 业务主线

- 选择：`D 双主线平台`
- 说明：交易执行与研究决策并列为两条核心业务主线，平台治理作为统一底座支撑。
- 影响：
  - 信息架构不能只围绕单一交易流程或单一研究流程展开。
  - 首页、导航、快捷入口、角色视角都需要同时支撑两条核心工作流。
  - 系统设计应让研究结果能够流向交易执行，同时保持两条主线可独立工作。

### 12. 技术演进策略

- 选择：`B 前端重点升级，后端稳住`
- 说明：优先升级前端架构、设计系统、状态模型和权限模型；后端服务边界总体保持稳定，逐步补齐统一鉴权、审计和配置能力。
- 影响：
  - 蓝图会把主要重构资源投入到 `web-portal` 的产品化与平台化升级。
  - 后端侧优先做统一鉴权上下文、权限校验、审计事件和管理配置收口，而不是立刻大规模改造服务边界。
  - 编码规划需要强调前端可维护性、跨端一致性和权限驱动渲染能力。

### 13. 视觉方向

- 选择：`B 专业平台 + 高级感`
- 说明：整体视觉需要兼顾专业可信度与平台级质感，在专业性基础上强化品牌感、层次感与体验完成度。
- 影响：
  - UI 蓝图应避免纯工具感的生硬界面，也避免泛 SaaS 化的轻飘风格。
  - 设计系统需要同时支持高信息密度页面和更强品牌表达的核心入口页。
  - 三端界面应体现统一视觉语言，但按终端特性调整节奏、层级与交互密度。

### 14. 移动端定位

- 选择：`B 交易与管理优先的移动工作台`
- 说明：移动端重点承接交易、风控、通知、审批、账户查看和轻量研究判断，不承载完整桌面级研究与复杂配置。
- 影响：
  - 三端蓝图应明确移动端不追求全量桌面能力，而是强调高频、短链路、可确认的关键任务。
  - 页面与权限设计需要支持移动端功能裁剪与场景聚焦。
  - 通知、审批、风控提醒、快速交易等能力应优先针对移动端优化。

### 15. 桌面端价值定位

- 选择：`B 专业工作台增强`
- 说明：桌面端应在 Web 基础上提供多窗口、常驻、快捷键、增强通知、盯盘与更沉浸的操作体验，而不是仅仅作为打包壳存在。
- 影响：
  - 三端蓝图需要区分“共享能力”和“桌面增强能力”。
  - 前端架构应预留桌面专属能力接入层，而不污染通用 Web 页面逻辑。
  - 设计系统和交互规范需要考虑多窗口协同、快捷键体系和持续监控场景。

### 16. AI 定位

- 选择：`C 强自动化代理`
- 说明：AI 不只是做分析或建议，还应主动触发策略、生成任务、推动执行链路，人工更多负责监督与干预。
- 影响：
  - 平台蓝图需要把 AI 视为核心工作流参与者，而不是附属功能模块。
  - 产品层应规划 AI 任务中心、代理状态、执行链路、人工接管和异常回退机制。
  - 技术层需要为 AI 编排、审批衔接、事件留痕和安全边界提供统一模型。

### 17. AI 自动执行边界

- 选择：`C AI 可按策略全自动执行`
- 说明：在满足策略与风控条件时，AI 可自动推进到执行层，人主要负责例外处理、监督与事后追踪。
- 影响：
  - 平台蓝图必须把自动执行、风控拦截、熔断机制和人工接管设计成核心基础能力。
  - 审批流将更多承担策略授权、风险授权与例外审查，而不是所有执行前的逐笔确认。
  - 编码与服务架构需要引入更明确的状态机、事件链路、执行日志和回滚/补偿机制。

### 18. AI 自动化最终兜底

- 选择：`C 多级熔断 + 分域急停开关 + 回滚/补偿`
- 说明：系统需要按交易、策略、权限、配置、任务等不同域分别熔断，并支持紧急停机、隔离、回滚与补偿。
- 影响：
  - 安全控制必须成为平台级底座，而不是单点业务规则。
  - 平台蓝图需要明确域级控制面板、熔断状态可视化、人工恢复流程和补偿链路。
  - 编码与架构层应采用可追踪的事件流、状态机与补偿机制，以支撑高自动化场景下的可控恢复。

### 19. 新旧系统迁移策略

- 修正结论：采用`独立重构工作区`，不再把长期双轨并行作为默认前提。
- 说明：
  - 重构应在单独文件夹中进行，所有工程文件都收敛在该目录内。
  - 现有项目可以停止运行，作为参考源与可复用资产来源保留。
  - 能复用的代码、配置、脚本和契约应复制到新工作区中，而不是继续直接依赖旧项目目录。
- 用户补充要求：新平台内部仍应支持手工开关控制。
- 影响：
  - 蓝图需要强调“独立建设、选择性复制、模块化启用”，而不是长期依赖旧系统并行。
  - 新平台内部仍应具备显式的开关/灰度控制能力，支持人工启用、人工回退与阶段性放量。
  - 目录结构、配置、构建链路和运行依赖都应以内聚的新工作区为准，避免跨目录耦合。

### 20. 开关控制粒度

- 选择：`B 模块 + 终端 + 角色级开关`
- 说明：开关能力需要支持按模块、终端和角色进行手工开启或关闭。
- 影响：
  - 灰度发布不应只按系统整体切换，而应支持更细的试点组合。
  - 前端菜单、路由入口、网关转发和配置中心需要共享同一套开关语义。
  - 平台后台应提供可视化开关管理，以支持运营式迁移和回退。

### 21. 工作台个性化程度

- 选择：`B 角色模板 + 用户可轻度定制`
- 说明：平台先提供按角色视角预设的标准工作台，用户可以调整常用卡片、快捷入口、默认排序和关注项，但不做完全自由编排。
- 影响：
  - 产品蓝图需要同时定义“系统级工作台模板”和“用户级偏好覆盖”。
  - 前端状态模型与配置存储需要支持轻量定制，而不引入过重的页面编排复杂度。
  - 三端需要共享工作台核心信息结构，但按终端提供不同密度和交互表达。

### 22. 通知与消息中心定位

- 选择：`B 核心行动中枢`
- 说明：通知中心不仅展示系统消息，还应承接审批、风控告警、AI 任务结果、异常接管和待办动作。
- 影响：
  - 消息中心需要从“通知列表”升级为“行动入口”。
  - 产品蓝图中应把通知、待办、审批、告警和异常接管打通。
  - 编码与架构层需要统一事件模型、消息投递和可执行动作协议。

### 23. 总体蓝图路径选择

- 选择：`B 双主线工作台优先，治理底座并行`
- 说明：以交易工作台和研究工作台作为产品正面，同时并行建设权限治理、审批审计、开关中心与消息行动中枢等平台底座。
- 影响：
  - 产品蓝图需要先把用户真正会进入的工作台做清楚，再把平台底座能力嵌进去。
  - 架构规划应优先服务前端产品化升级，同时为后端逐步补齐统一能力提供收口点。
  - 项目推进方式应避免“先做纯后台治理、用户感知很弱”或“只做炫酷前端、底座空心化”两种失衡路径。

## 当前形成的核心原则

1. `Refactor V2` 应定位为多角色、多视角、三端统一的交易与研究平台。
2. 权限系统是产品骨架，菜单系统必须由权限与视角驱动。
3. 产品架构必须兼顾交易、研究、运营管理三类主要工作流。
4. 本轮蓝图应先服务单企业平台落地，同时避免阻断后续多租户演进。

- 第一阶段蓝图中哪些域是必须优先成型的。

## 已确认设计草案

### A. 产品定位与信息架构

- 已确认：`Refactor V2` 应被定义为多角色、多视角、三端统一、AI 驱动的交易与研究平台。
- 已确认：平台具有两条并列主线：
  - `交易执行`
  - `研究决策`
- 已确认：平台治理作为统一底座持续存在，承载权限、审批、审计、开关、消息行动中枢和自动化安全控制。
- 已确认：一级信息架构建议固定为 5 个域：
  - `交易域`
  - `研究域`
  - `行动域`
  - `治理域`
  - `系统域`
- 已确认：视角不等于另一套产品，而是同一套平台在不同角色下的入口编排：
  - `交易视角`：突出交易域与行动域
  - `研究视角`：突出研究域与行动域
  - `管理视角`：突出治理域与系统域
  - `综合视角`：面向超级管理员或高权限用户

### B. 角色体系、权限模型与菜单模型

- 已确认：权限体系不应简化为“角色表 + 菜单表”，而应拆为 6 层：
  - `组织层`
  - `身份层`
  - `角色层`
  - `权限层`
  - `视角层`
  - `开关层`
- 已确认：角色应为可配置实体，而不是写死枚举。
- 已确认：权限至少覆盖 4 类能力：
  - `菜单权限`
  - `页面权限`
  - `操作权限`
  - `数据范围权限`
- 已确认：视角不是权限别名，而是角色组合后的工作台编排器。
- 已确认：菜单模型不应写死在前端路由中，而应由四段式模型驱动：
  - `平台菜单清单`
  - `角色授权`
  - `终端适配`
  - `视角编排`
- 已确认：开关层应叠加在权限层之上，支持按模块、终端、角色进行手工启停。

### C. 工作台体系与关键用户流

- 已确认：产品首页应升级为“视角工作台”，而不是所有角色共用的大总览。
- 已确认：各视角工作台应遵循统一骨架：
  - `顶部`：当前视角、账户/角色切换、全局搜索/命令入口、关键状态、快速动作
  - `中部左侧`：当前视角最重要的业务概览
  - `中部右侧`：行动流，承接消息、审批、AI 任务结果、异常接管和建议动作
  - `底部`：个性化区，放常用模块、快捷入口、关注列表和最近访问
- 已确认：平台需要围绕 4 条主链设计：
  - `交易链`
  - `研究链`
  - `治理链`
  - `自动化链`
- 已确认：研究结果、AI 结论和风险提示应直接汇入交易链，而不是散落在独立页面中。
- 已确认：AI 自动化链需要成为产品主叙事之一，而不是隐藏在单个 AI 页面中。
- 已确认：三端映射策略为：
  - Web：承载完整工作流
  - 桌面端：强化多窗口、盯盘、快捷键和持续监控
  - 移动端：聚焦行动流、审批、告警、通知、风控确认、账户查看和快速交易

### D. AI、消息中枢、审批审计与安全控制底座

- 已确认：平台差异化核心不应是“多几个 AI 页面”，而应是把 AI、消息、审批、风控和自动化执行编成统一控制网。
- 已确认：建议围绕统一的 `Action` 对象建模，将 AI 建议、审批单、自动执行、异常告警等收敛为可追踪、可分派、可执行、可回退的行动对象。
- 已确认：底座需要包含 4 个中枢：
  - `AI 代理中枢`
  - `消息行动中枢`
  - `审批审计中枢`
  - `安全控制中枢`
- 已确认：AI 代理需要作为平台级执行参与者存在，而不是单一页面功能。
- 已确认：消息中心应成为统一行动入口，承接审批、待办、告警、AI 结果、人工接管和恢复建议。
- 已确认：审批不一定拦截所有动作，但必须覆盖规则、边界和例外治理。
- 已确认：安全控制应支持：
  - 多级熔断
  - 分域急停开关
  - 紧急停机
  - 人工恢复
  - 补偿和回滚
- 已确认：平台运行逻辑建议统一为：
  - AI 发现机会或异常，产生 `Action`
  - `Action` 进入消息行动中枢
  - 系统根据策略、权限、风控规则决定自动执行、进入审批，或直接熔断
  - 全链路进入审计
  - 若失败或超阈值，转入异常接管与补偿流程

### E. UI、设计系统与三端体验蓝图

- 已确认：整体视觉方向应为“专业平台 + 高级感”，同时传达：
  - `可信`
  - `高控制力`
  - `系统正在运行`
- 已确认：UI 系统建议拆为 5 层：
  - `Shell 层`
  - `工作台层`
  - `Module 层`
  - `Action 层`
  - `Feedback 层`
- 已确认：Web 和桌面端允许更高信息密度，移动端必须强化分组、层级和行动优先。
- 已确认：三端职责划分为：
  - Web：主平台端，承接完整链路
  - Desktop：强化多窗口、盯盘常驻、快捷键、系统级通知和沉浸式工作流
  - Mobile：定位为行动终端，而非缩小版后台
- 已确认：应优先沉淀一批平台级组件：
  - `ViewSwitcher`
  - `WorkspaceHero`
  - `ActionFeed`
  - `RiskBanner`
  - `ApprovalDrawer`
  - `AgentStatusPanel`
  - `FeatureFlagPanel`
  - `AuditTimeline`
  - `CommandPalette`
  - `QuickTradePanel`
- 已确认：设计目标可概括为：
  - 让复杂系统看起来有秩序
  - 让高风险操作看起来可控
  - 让自动化能力看起来可信
